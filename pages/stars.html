What Is STARS

Every regulated industry shares the same underlying problem. STARS solves it once.

The core problem in regulated environments isn't capability — it's translation. Physical systems generate data. Regulators require structured evidence. Between those two states, there is a gap that organizations fill with manual documentation, tribal knowledge, and post-hoc reconstruction — all of which fail audits.

STARS (Secure Translational Automated Regulatory System) is the architecture that closes that gap. It is a schema — a structured data model — that sits between the physical world and the compliance world. It ingests operational data from sensors, systems, and processes; normalizes it through eight atomic tables; and passes it through a deterministic rule engine that outputs framework-specific compliance artifacts.

STARS is embedded in TRIDENT as an integrated layer and available as the underlying architecture for AURUM's electrochemical sensing output. The same schema serves every vertical — Defense, Aerospace, Healthcare, Critical Minerals, Maritime — because STARS is agnostic to the regulatory framework. Only the catalog content and rule sets change between industries.

What STARS Is Not

✕Not a compliance software platform

STARS does not help organizations manage compliance. It generates evidence that compliance has occurred.

✕Not a GRC tool

STARS does not track risk, assign controls, or manage policies. It translates physical events into regulatory evidence artifacts.

✕Not framework-specific

STARS serves CMMC, AS9100D, HIPAA, IMO, and others from the same architecture. New frameworks are catalog updates, not platform rebuilds.

The Schema

Eight atomic tables that normalize any physical event into compliance language.

Every compliance requirement — across every framework Aegis supports — can be expressed as a combination of these eight dimensions. The tables are universal; what changes between industries is the catalog content that maps each dimension to a specific regulatory obligation.

Table 01

WHO

Who performed this action?

Identity, role, authorization level, and access credentials of the person or system performing the documented action. Required for CMMC access control evidence, AS9100D personnel qualification records, and HIPAA authorization trails.

Identity · Role · Authorization · Credentials

Table 02

WHAT

What was done or affected?

Asset identifier, action classification, and object state. Anchors the compliance record to a specific item, system, or data element. Supports ITAR-controlled item tracking, AS9100D part traceability, and CUI object identification.

Asset ID · Action Type · Object State

Table 03

WHEN

When did this occur?

Timestamp, duration, sequence index, and retention classification. Creates the time-ordered audit trail required by CMMC audit logging controls, FDA 21 CFR Part 11, and IMO incident reporting timelines.

Timestamp · Duration · Sequence · Retention

Table 04

WHERE

Where did this occur?

Location, facility zone, environmental parameters, and jurisdictional classification. Required for ITAR geographic controls, AS9100D process environment documentation, and CTPAT facility security evidence.

Location · Zone · Environment · Jurisdiction

Table 05

WHY

What requirement triggered this?

The regulatory requirement, standard clause, or control that mandates the documented action. Directly links the physical event to the specific obligation it satisfies — the key to generating pre-structured, assessor-ready artifacts.

Requirement · Clause · Control ID · Standard

Table 06

HOW

How was this performed?

Method, procedure reference, tooling, and configuration. Documents the process by which the action was performed — supporting AS9100D process control evidence, CMMC configuration management records, and FDA procedure compliance documentation.

Method · Procedure · Tool · Config

Table 07

HOW MANY

To what quantity or threshold?

Quantity, measurement value, tolerance, and threshold comparison. Enables TRIDENT's quantitative QA scoring to appear as structured compliance data, and supports AURUM's electrochemical concentration readings as provenance verification evidence.

Quantity · Measurement · Tolerance · Threshold

Table 08

META

What is the provenance of this record?

Schema version, data source identifier, chain-of-custody hash, and record integrity signature. Ensures every compliance artifact carries cryptographic provenance — required for CMMC audit integrity and DFARS/CUI data handling evidence.

Version · Source · Hash · Integrity

The Rule Engine

8-Ball: deterministic by design.

Once STARS has structured incoming data through its eight atomic tables, the 8-Ball rule engine evaluates that structured output against the active regulatory catalog and produces framework-specific compliance artifacts.

The word "deterministic" is doing real work here. Given the same structured input and the same regulatory catalog, 8-Ball always produces the same output. There is no inference, no probabilistic matching, no AI interpretation. The same physical event, documented the same way, maps to the same compliance evidence every time.

This matters because assessors and auditors are not evaluating AI-generated summaries — they are examining specific controls, specific clauses, and specific evidence trails. Deterministic output means Aegis's evidence packages survive adversarial review without interpretation variance.

8-Ball Processing Flow

01 Ingest structured table output
Receives the eight populated STARS tables from the sensing or ingestion layer.
02 Load active regulatory catalog
Retrieves the framework-specific rule set for the active compliance domain.
03 Map to control or clause
Deterministically maps each table field to the specific regulatory requirement it satisfies.
04 Output evidence artifact
Produces a structured, framework-specific document organized to the exact assessor expectation.

Why STARS

What makes a deterministic schema different from every other compliance approach.

01

No Interpretation Drift

Deterministic rule processing means the same physical event maps to the same compliance evidence every time. No variance between assessments, no assessor-specific interpretation, no documentation that describes an event differently than how it occurred.

02

Framework-Agnostic by Design

The eight atomic tables normalize events in a way that is independent of any specific regulatory standard. Adding a new framework to STARS is a catalog and rule set update — the schema itself does not change. This is what makes STARS a platform, not a solution.

03

Evidence at the Point of Activity

STARS generates compliance documentation at the moment the physical event occurs — not after the fact, not from memory, not from manual reconstruction. The temporal integrity of the evidence chain is built into the architecture.

04

Cryptographic Provenance

The Meta table carries a chain-of-custody hash and record integrity signature for every artifact STARS produces. Evidence packages are verifiably traceable to the physical event that generated them — from sensor reading to assessor submission.

05

Simultaneous Multi-Framework Output

A single physical event can simultaneously satisfy requirements across multiple frameworks — CMMC and AS9100D and ITAR from the same capture. Organizations with overlapping compliance obligations do not run parallel documentation workflows.

06

Sensor-Agnostic Ingestion

STARS ingests structured data from any compliant source — TRIDENT's hardware platform, AURUM's electrochemical sensing substrate, or third-party sensor systems with a compatible output format. The schema defines the structure; the source is variable.
